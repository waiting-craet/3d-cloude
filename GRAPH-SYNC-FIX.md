# 图谱数据同步功能修复

## 问题描述

当用户点击"修改"按钮进入2D编辑页面，对知识图谱进行修改后，点击"保存并转换为3D"按钮时，系统会创建新的节点，而不是更新现有的节点。

## 根本原因

原来的 `/api/convert` API 总是创建新节点，没有检查节点是否已经存在于数据库中。

## 解决方案

实现了智能同步功能，系统现在会：

1. **检测变更**：比较2D工作流数据和数据库数据
2. **更新现有节点**：对已存在的节点进行更新（保留ID）
3. **添加新节点**：只为真正新增的节点创建数据库记录
4. **删除节点**：移除在编辑中被删除的节点
5. **同步边**：同样智能地处理边的添加、更新和删除

## 实现的文件

### 1. 变更检测模块 (`lib/graph-sync.ts`)

提供函数来检测工作流数据和数据库数据之间的差异：

- `detectNodeChanges()` - 识别添加、修改、删除的节点
- `detectEdgeChanges()` - 识别添加、修改、删除的边
- `detectChanges()` - 综合检测所有变更

### 2. 同步API (`app/api/graphs/[id]/sync/route.ts`)

新的API端点 `POST /api/graphs/[id]/sync`：

- 获取现有图谱数据
- 检测变更
- 在事务中执行所有同步操作
- 返回详细的统计信息

### 3. WorkflowCanvas组件更新 (`components/WorkflowCanvas.tsx`)

修改 `saveAndConvert()` 函数：

- 保留节点和边的数据库ID
- 调用 `/api/graphs/[id]/sync` 而不是 `/api/convert`
- 显示同步统计信息

## 工作流程

```
用户编辑2D图谱
    ↓
点击"保存并转换为3D"
    ↓
WorkflowCanvas.saveAndConvert()
    ↓
调用 /api/graphs/[id]/sync
    ↓
1. 获取数据库中的现有数据
2. 检测变更（添加/更新/删除）
3. 在事务中执行同步操作
   - 删除被移除的节点和边
   - 更新修改的节点和边
   - 添加新的节点和边
4. 更新图谱和项目的统计计数
    ↓
返回3D视图（显示更新后的图谱）
```

## 性能优化

### 1. 批量操作

所有数据库操作都使用 `Promise.all()` 并行执行：

- 节点更新：并行更新所有修改的节点
- 节点创建：并行创建所有新节点
- 边更新：并行更新所有修改的边
- 边创建：并行创建所有新边

### 2. 事务超时配置

```typescript
await prisma.$transaction(async (tx) => {
  // ... 同步操作
}, {
  maxWait: 10000, // 最大等待时间 10秒
  timeout: 15000, // 事务超时时间 15秒
})
```

这确保了即使处理大量数据也不会超时。

## 关键特性

### 1. ID保留

- 现有节点保留其数据库ID
- 更新操作不会创建重复节点

### 2. 引用完整性

- 删除节点时自动删除相关的边
- 确保没有孤立的边

### 3. 事务安全

- 所有操作在单个事务中执行
- 如果任何操作失败，所有更改都会回滚

### 4. 统计准确性

- 自动更新图谱的节点和边计数
- 同时更新项目级别的统计

## 测试步骤

1. **创建初始图谱**
   - 在2D视图创建几个节点和连接
   - 保存并转换为3D
   - 记录节点数量

2. **编辑现有图谱**
   - 点击"修改"按钮
   - 修改现有节点的标题或描述
   - 添加新节点
   - 删除某些节点
   - 保存并转换为3D

3. **验证结果**
   - ✅ 修改的节点应该被更新（不是创建新的）
   - ✅ 新节点应该被添加
   - ✅ 删除的节点应该消失
   - ✅ 节点总数应该正确
   - ✅ 没有重复的节点

## 注意事项

1. **videoUrl字段**：数据库schema中没有`videoUrl`字段，已从同步逻辑中移除
2. **坐标转换**：新节点会自动计算3D坐标
3. **类型安全**：所有类型错误已修复
